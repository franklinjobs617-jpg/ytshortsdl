# Payment Recovery Playbook（支付未完成挽回方案）

> 适用项目：`ytshortsdl`  
> 目标：用户发起支付但未成功时，自动触发邮件挽回，并支持优惠券转化。  
> 状态：规划文档（后续按步骤开发）。

---

## 1. 目标与范围

- 提升支付完成率（重点提升 Stripe / PayPal 的 checkout -> paid 转化）。
- 降低“发起很多、实际付款很少”的漏斗损耗。
- 提供可追踪、可回滚、可控风控的邮件挽回机制。

---

## 2. 服务商选择（建议）

## 推荐：Resend

- 原因：
  - API 简单、开发快、对 Next.js/Node 友好。
  - 3,000 封/月足够做支付挽回场景。
  - 模板与投递日志易维护。

## 备选

- Brevo：免费额度大（300/天），但集成复杂度略高。
- MailerSend：可用，体验中规中矩。
- Gmail SMTP：仅适合测试，不建议生产（封号风险高）。

---

## 3. 统一业务口径（必须先定）

- Stripe 未完成：`pay.status = "0"`（待支付）。
- PayPal 未完成：`paypal_pay.status = "1"`（待支付）。
- Stripe 成功：`pay.status = "1"`。
- PayPal 成功：`paypal_pay.status = "2"`。

> 注意：Stripe 与 PayPal 状态值不一致，查询逻辑必须分开处理。

---

## 4. “支付未完成”判定规则

满足以下条件才进入挽回池：

1. 存在支付发起记录（有 checkout 链接 / orderId）。
2. 发起后超过 30 分钟仍未成功。
3. 未被标记为已挽回完成（防重复触达）。
4. 用户邮箱有效且未退订营销邮件。

---

## 5. 触达节奏（建议三封）

- T+30分钟：提醒邮件（无优惠）。
- T+24小时：第二封（含优惠券）。
- T+72小时：最后召回（限时文案）。

每个阶段每个订单只发一次。

---

## 6. 优惠策略（Stripe + PayPal）

## Stripe（官方推荐做法）

1. 创建 `Coupon`（例如 10% off）。
2. 创建 `Promotion Code`（用户可输入的优惠码）。
3. 邮件发送优惠码给目标用户。
4. Checkout 时携带 promotion code，或让用户手动输入。

## PayPal（建议业务层优惠码）

PayPal 侧不走 Stripe 那种统一 Promotion Code 逻辑时，建议：

1. 系统生成业务优惠码（例如 `COME_BACK_10`）。
2. 前端支付页支持输入优惠码。
3. 后端校验优惠码后，计算折后价再创建 PayPal order/subscription。

---

## 7. 数据设计（最小必需）

建议新增（或扩展）一张挽回记录表，例如 `payment_recovery_log`：

- `id`
- `provider`（stripe/paypal）
- `order_id` / `session_id`
- `user_id`
- `email`
- `stage`（1/2/3）
- `status`（scheduled/sent/failed/converted）
- `coupon_code`（可空）
- `sent_at`
- `error_message`
- `created_at` / `updated_at`

以及用户退订字段（已有用户表可直接扩展）：

- `email_opt_out`（0/1）

---

## 8. 定时任务流程（每小时执行）

1. 拉取“未支付且达到时间窗口”的订单。
2. 过滤已发送该阶段邮件的记录。
3. 生成（或获取）对应阶段模板与优惠码。
4. 调用 Resend API 发送邮件。
5. 落库发送结果（成功/失败）。
6. 若后续支付成功，更新挽回记录状态为 `converted`。

---

## 9. 追踪指标（上线后必须看）

- 发起支付数
- 到达收银台数
- 支付成功数
- 挽回邮件发送数
- 邮件打开率 / 点击率
- 优惠码使用率
- 挽回转化率（邮件后24h内成功支付）

---

## 10. 风险清单与防护

## 风险 1：重复发邮件骚扰用户
- 防护：`order_id + stage` 唯一约束；发送前先查 log。

## 风险 2：优惠码被滥用
- 防护：优惠码绑定用户、单次使用、设置有效期（如 7 天）。

## 风险 3：状态误判导致“误发挽回”
- 防护：严格使用支付状态接口，不用“积分变化”作为主判定。

## 风险 4：支付成功后仍继续发提醒
- 防护：每轮任务先拉最新支付状态，已 paid 直接跳过并标记 converted。

## 风险 5：合规问题（邮件投诉）
- 防护：提供退订入口，尊重 `email_opt_out`。

---

## 11. 实施顺序（建议）

### Phase 1（先跑通）
- 接入 Resend
- 建 recovery log 表
- 上线 T+30min 第一封（无优惠）
- 跑一个小流量灰度

### Phase 2（提转化）
- 增加 T+24h 优惠券邮件
- Stripe promotion code 接入
- PayPal 业务优惠码接入

### Phase 3（优化）
- 增加 T+72h 最后一封
- 埋点仪表盘与 A/B 测试（标题、折扣、文案）

---

## 12. 验收标准（Definition of Done）

- 能稳定识别“未支付订单”。
- 同一订单同一阶段不会重复发送邮件。
- Stripe/PayPal 都能完成优惠后支付流程。
- 用户支付成功后，前端显示与后端状态一致。
- 指标面板能看到挽回邮件带来的新增支付转化。
